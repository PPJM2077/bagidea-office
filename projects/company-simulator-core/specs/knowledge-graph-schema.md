# Knowledge Graph Schema

> **Status:** Draft  
> **Owner:** Research Director  
> **Version:** 0.1.0  
> **Store:** Neo4j (primary) or PostgreSQL recursive CTEs (lightweight fallback)  
> **Integration:** Event-to-Entity pipeline reads NATS `sim.event.*` → creates/updates graph entities  

---

## 1. Purpose

Define the **node types, edge types, and property schemas** for the AI Quant Office Knowledge Graph.  

The Knowledge Graph captures **entities and their relationships** — who reports to whom, what decisions caused what events, which strategies are owned by which agents, etc. It enables:
- Multi-hop reasoning ("who has authority to approve this?")
- Org traversal ("show the chain of command for this department")
- Causal tracing ("what decisions led to this outcome?")
- Hybrid retrieval (graph + vector — query time fusion)

## 3. Edge / Relationship Types

### 3.1 Relationship Catalog

| Relationship | From | To | Description | Cardinality |
|-------------|------|----|-------------|-------------|
| `REPORTS_TO` | `Agent` | `Agent` | Reporting line (manager → subordinate) | N:1 |
| `BELONGS_TO` | `Agent` | `Department` | Department membership | N:1 |
| `MADE_DECISION` | `Agent` | `Decision` | Agent authored a decision | N:M |
| `CAUSED` | `Decision` | `Event` | Decision triggered an event | 1:N |
| `AFFECTED` | `Event` | `Company` | Event impacted company state | N:1 |
| `AFFECTED` | `Event` | `Department` | Event impacted department | N:1 |
| `OWNED_BY` | `Strategy` | `Agent` | Strategy is owned by an agent | N:1 |
| `IDENTIFIED_BY` | `Risk` | `Agent` | Risk was flagged by an agent | N:1 |
| `REFERENCED_BY` | `Decision` | `KnowledgeDoc` | Decision cited a document as evidence | N:M |
| `REFERENCED_BY` | `Decision` | `Event` | Decision cited an event as evidence | N:M |
| `COLLABORATES_WITH` | `Agent` | `Agent` | Agents worked together on a task | N:M |
| `APPROVES` | `Agent` | `Decision` | Agent approved/reviewed a decision | N:M |
| `DEPENDS_ON` | `Strategy` | `Strategy` | A strategy depends on another | N:M |
| `DEPENDS_ON` | `Department` | `Department` | Inter-department dependency | N:M |
| `PART_OF` | `Decision` | `Strategy` | Decision was made as part of a strategy | N:1 |
| `TRIGGERED` | `Event` | `Risk` | Event triggered a risk breach | N:1 |
| `LEADS` | `Agent` | `Department` | Agent is head of department | N:1 (unique) |

### 3.2 Relationship Properties

Some relationships carry properties that provide context:

```cypher
// Reporting line with time bounds
(:Agent)-[:REPORTS_TO {
  since_tick: 100,
  until_tick: null,           // null = current
  role_at_time: "junior_quant",
  relationship: "solid"       // solid | dotted (matrix org)
}]->(:Agent)

// Collaboration with weight (could be inferred from co-editing, co-deciding)
(:Agent)-[:COLLABORATES_WITH {
  strength: 0.75,             // 0.0-1.0, based on frequency/recentcy
  first_interaction_tick: 500,
  last_interaction_tick: 1040,
  interaction_count: 47
}]->(:Agent)

// Decision approval (audit trail)
(:Agent)-[:APPROVES {
  at_tick: 1042,
  role_at_time: "cto",
  approved: true,
  comments: "Approved with minor conditions"
}]->(:Decision)
```

## 5. Entity Extraction Pipeline

The Knowledge Graph is **not manually curated**. It grows automatically from the event stream.

### 5.1 Pipeline Architecture

```
NATS Subject: sim.event.*  ──┬── Event-to-Entity Consumer
                              │
                              ▼
                     ┌──────────────────┐
                     │  NATS Consumer   │  (app/workers/entity_extractor.py)
                     │  (Queue Group)   │
                     └────────┬─────────┘
                              │
                              ▼
                     ┌──────────────────┐
                     │  Entity Router   │
                     │  Classify event  │
                     │  type → entity   │
                     └────────┬─────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
      ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
      │ Agent Entity  │ │ Decision     │ │ Event Entity │
      │ Extractor    │ │ Extractor    │ │ Extractor    │
      └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
             │                │                │
             ▼                ▼                ▼
      ┌─────────────────────────────────────────────┐
      │          Entity Resolver & Deduplicator      │
      │  - Check if entity exists (by ID / content) │
      │  - Merge properties if exists                │
      │  - Create or update node/edge                │
      └─────────────────────┬───────────────────────┘
                            │
                            ▼
      ┌─────────────────────────────────────────────┐
      │          Graph Writer                        │
      │  - Cypher MERGE (upsert)                     │
      │  - Create relationships                      │
      │  - Update timestamps                         │
      └─────────────────────────────────────────────┘
```

### 5.2 Event-to-Entity Mapping Rules

| Event Type | Node(s) Created | Edge(s) Created |
|------------|-----------------|-----------------|
| `sim.employee.hired` | Agent (if new) | `(Agent)-[:BELONGS_TO]->(Department)`, `(Agent)-[:REPORTS_TO]->(Manager)` |
| `sim.employee.terminated` | — | Update Agent status; close `REPORTS_TO` edge with `until_tick` |
| `sim.employee.promoted` | — | Update `REPORTS_TO` edge `until_tick`, create new `REPORTS_TO` |
| `sim.decision.made` | Decision | `(Agent)-[:MADE_DECISION]->(Decision)`, `(Decision)-[:PART_OF]->(Strategy)` |
| `sim.decision.executed` | Event | `(Decision)-[:CAUSED]->(Event)`, `(Event)-[:AFFECTED]->(Company)` |
| `sim.risk.breached` | Risk (if new) | `(Event)-[:TRIGGERED]->(Risk)`, `(Risk)-[:IDENTIFIED_BY]->(Agent)` |
| `sim.strategy.created` | Strategy | `(Strategy)-[:OWNED_BY]->(Agent)` |
| `sim.strategy.promoted` | — | Update `Strategy.lifecycle_stage` |
| `sim.collaboration.started` | — | `(Agent)-[:COLLABORATES_WITH]->(Agent)` with strength updated |
| `sim.knowledge.indexed` | KnowledgeDoc | `(KnowledgeDoc)-[:REFERENCED_BY]->(Decision)` if linked |

### 5.3 Entity Resolution

```
FUNCTION upsert_entity(label, properties):
    // Try to find existing node by ID
    existing = MATCH (n:{label} {id: properties.id}) RETURN n

    IF existing:
        // Update properties (merge, not replace)
        MERGE (n:{label} {id: properties.id})
        SET n += properties.properties_to_update
        SET n.updated_at_tick = current_tick
        RETURN n
    ELSE:
        // Create new node
        CREATE (n:{label})
        SET n = properties
        SET n.created_at_tick = current_tick
        SET n.updated_at_tick = current_tick
        RETURN n

FUNCTION upsert_relationship(from_id, to_id, type, properties):
    MERGE (from {id: from_id})-[r:{type}]->(to {id: to_id})
    SET r += properties
    SET r.last_seen_tick = current_tick
    RETURN r
```

```
│  │  • Entities       │  │  • Event vectors  │  │  • Entity nodes      │  │

│  │  • State          │  │  • Doc embeddings │  │  • Hierarchy         │  │
│  │  • Audit trail    │  │  • Similarity     │  │  • Causality         │  │

│  ┌──────────────────────────────────────────────────────────────────┐   │
│  │                  HYBRID RETRIEVAL ROUTER                          │   │
│  │  Parse query → classify intent → vector OR graph OR both → fuse  │   │
│  └──────────────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────────────┘
```

### 6.2 Query Routing Logic

```
FUNCTION hybrid_retrieve(query, context):
    intent = CLASSIFY_QUERY_INTENT(query)
    // Returns: "semantic" | "structural" | "temporal" | "hybrid"

    SWITCH intent:
        CASE "semantic":
            // "What happened in the last meeting about risk?"
            RETURN vector_search(query, qdrant, filter={...})

        CASE "structural":
            // "Who reports to the CRO?"
            RETURN graph_query(query, neo4j, cypher="...")

        CASE "temporal":
            // "What decisions were made between tick 800 and 900?"
            RETURN graph_query(query, neo4j, temporal_filter=true)

        CASE "hybrid":
            // "What risks emerged in the trading division recently?"
            vector_results = vector_search(query, qdrant, ...)
            graph_results = graph_query(query, neo4j, ...)
            RETURN fuse_results(vector_results, graph_results,
                                strategy="reciprocal_rank_fusion")
```

### 6.3 Cross-Store References

Every graph node stores:
- `embedding_id: "qdrant:point_xxx"` — link to the semantic vector in Qdrant
- `pg_id: "uuid:..."` — link to the PostgreSQL row (if applicable)

This enables cross-store traversal:
```cypher
// Find a decision node, then fetch its full text from Qdrant
MATCH (d:Decision {id: "dec_01J7..."})
RETURN d.id, d.summary, d.embedding_id
// → Use embedding_id to fetch full text vector from Qdrant
```

## 7. Tier Enforcement (Knowledge Pyramid)

The blueprint defines a 5-tier Knowledge Pyramid (T0-T4). Every `KnowledgeDoc` node carries a `tier` property.

| Tier | Label | Color | Access | Freshness Requirement | Source |
|------|-------|-------|--------|-----------------------|--------|
| T0 | Immutable | 🔴 Red | All agents read, no agent writes | Never stale | Constitution, risk policy |
| T1 | Official | 🟠 Orange | All agents read, only authorized write | 1 tick | API docs, broker specs |
| T2 | Internal | 🟡 Yellow | Department agents read, owner writes | 10 ticks | Research, backtest, journal |
| T3 | External | 🟢 Green | All agents read | 50 ticks | Papers, blogs, GitHub |
| T4 | Community | 🔵 Blue | Reference only, not used for decisions | 100 ticks | Reddit, Discord, X |

**Enforcement in graph queries:**
```cypher
// Agent queries knowledge — automatically filtered by tier
MATCH (a:Agent {id: $agent_id})-[:BELONGS_TO]->(d:Department)
MATCH (k:KnowledgeDoc)
WHERE k.tier <= $max_tier_allowed   // per agent's role and security clearance
  AND (k.tier <= 1                  // T0, T1: always readable
       OR k.tier <= 2 AND k.source IN $dept_knowledge_sources
       OR k.tier >= 3)              // T3, T4: readable by all
RETURN k
```

## 8. Example Graph Queries

### 8.1 Org Chart Traversal

```cypher
// Find the full reporting chain from an employee to the CEO
MATCH path = (emp:Agent {id: "agent:quant_042"})-[:REPORTS_TO*]->(ceo:Agent {role: "ceo"})
RETURN [node IN nodes(path) | node.id] AS chain,
       length(path) AS depth
```

### 8.2 Authority Check

```cypher
// Who can approve a $500k budget request?
// (Find agents with approval authority in finance)
MATCH (a:Agent)
WHERE "financial_approval" IN a.capabilities
  AND a.priority_level >= 3
  AND a.status = "online"
RETURN a.id AS approver, a.role,
       CASE WHEN a.role IN ["ceo", "cro"] THEN "unlimited"
            WHEN a.role = "cfo" THEN "up_to_1M"
            ELSE "up_to_100K"
       END AS approval_limit
```

### 8.3 Causal Tracing

```cypher
// Trace: "What chain of decisions and events led to the current cash crisis?"
MATCH path = (cause:Decision)-[:CAUSED]->(evt:Event)-[:AFFECTED]->(c:Company {id: $company_id})
MATCH (risk:Risk {metric: "cash_runway", status: "active"})
WHERE evt.tick >= risk.tick_detected - 50
RETURN cause.id AS decision_id,
       cause.summary AS decision,
       evt.summary AS event,
       evt.tick AS at_tick
ORDER BY evt.tick ASC
```

### 8.4 Strategy Lifecycle Audit

```cypher
// Show all decisions made as part of a strategy
MATCH (s:Strategy {id: "strat_01J7..."})
MATCH (d:Decision)-[:PART_OF]->(s)
OPTIONAL MATCH (maker:Agent)-[:MADE_DECISION]->(d)
OPTIONAL MATCH (approver:Agent)-[:APPROVES]->(d)
RETURN d.tick, d.summary, d.status,
       maker.id AS decision_maker,
       collect(DISTINCT approver.id) AS approvers
ORDER BY d.tick
```

### 8.5 Collaboration Network

```cypher
// Find the most collaborative agents (for assigning cross-functional tasks)
MATCH (a:Agent)-[c:COLLABORATES_WITH]->(other:Agent)
WHERE c.strength > 0.5
RETURN a.id AS agent,
       count(other) AS collaborators,
       avg(c.strength) AS avg_strength
ORDER BY collaborators DESC
LIMIT 10
```

### 8.6 Risk Correlation

```cypher
// Which risks tend to co-occur? (Useful for predicting cascading failures)
MATCH (r1:Risk {status: "active"})<-[:TRIGGERED]-(e:Event)-[:TRIGGERED]->(r2:Risk)
WHERE r1.id <> r2.id
RETURN r1.category AS risk_category,
       r2.metric AS related_risk,
       count(*) AS co_occurrence_count
ORDER BY co_occurrence_count DESC
```

## 10. Graph Store Decision Matrix

| Consideration | Neo4j (self-host) | PostgreSQL Recursive CTEs |
|---------------|-------------------|---------------------------|
| **Setup time** | 1-2 days | 0 (already have PG) |
| **Query syntax** | Cypher (graph-native) | SQL WITH RECURSIVE |
| **Multi-hop depth** | Any depth, constant time | O(depth), slows significantly past 5 hops |
| **Property graph model** | Native | Simulated (node + edge tables) |
| **Scale limit** | 10B+ nodes | ~100K nodes before perf degrades |
| **Operations overhead** | New cluster, backups | Zero — uses existing PG |
| **GraphRAG integration** | Native (APOC, GDS) | Not possible |
| **Community detection** | Built-in (Louvain, Label Prop) | Custom implementation needed |
| **Best for** | Production with 10K+ nodes | Phase 0 prototype, small graphs |

**Recommendation:** Start with PostgreSQL recursive CTEs (zero new infra, test within existing schema). Migrate to Neo4j when the graph exceeds 10K nodes or when multi-hop queries need to go deeper than 5 levels.

## 11. Appendix: PostgreSQL Recursive CTE Implementation

For the fallback PG-graph option, use these tables:

```sql
-- Graph nodes (supertype for all node types)
CREATE TABLE kg_nodes (
    id              TEXT PRIMARY KEY,
    label           TEXT NOT NULL,          -- Agent | Company | Department | Decision | ...
    properties      JSONB NOT NULL DEFAULT '{}',
    embedding_id    TEXT,                    -- link to Qdrant point
    created_at_tick BIGINT NOT NULL,
    updated_at_tick BIGINT NOT NULL
);

CREATE INDEX idx_kg_nodes_label ON kg_nodes(label);

-- Graph edges (relationships)
CREATE TABLE kg_edges (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_id       TEXT NOT NULL REFERENCES kg_nodes(id),
    target_id       TEXT NOT NULL REFERENCES kg_nodes(id),
    type            TEXT NOT NULL,          -- REPORTS_TO | BELONGS_TO | CAUSED | ...
    properties      JSONB NOT NULL DEFAULT '{}',
    created_at_tick BIGINT NOT NULL,
    UNIQUE (source_id, target_id, type)
);

CREATE INDEX idx_kg_edges_source ON kg_edges(source_id);
CREATE INDEX idx_kg_edges_target ON kg_edges(target_id);
CREATE INDEX idx_kg_edges_type ON kg_edges(type);

-- Multi-hop query (example: org chain)
WITH RECURSIVE org_chain AS (
    -- Base: start employee
    SELECT n.id, n.properties->>'name' AS name, 0 AS depth
    FROM kg_nodes n
    WHERE n.id = 'agent:quant_042'

    UNION ALL

    -- Recursive: find manager
    SELECT n.id, n.properties->>'name' AS name, oc.depth + 1
    FROM org_chain oc
    JOIN kg_edges e ON e.source_id = oc.id AND e.type = 'REPORTS_TO'
    JOIN kg_nodes n ON n.id = e.target_id
)
SELECT * FROM org_chain ORDER BY depth;
```
